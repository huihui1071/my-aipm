模块1：LLM与AI技术基础

面试高频指数：★★★★★

你的掌握程度：★★★（需重点加强）

▾ Q1：Transformer架构（必考）

Transformer = Encoder + Decoder，核心创新是自注意力机制（Self-Attention）。

Self-Attention计算过程：

  • 输入向量通过三个线性变换分别得到 Q（Query）、K（Key）、V（Value） 矩阵
  • 公式：Attention(Q,K,V) = softmax(QK^T / √d_k) × V
  • 直觉：Q是"我在找什么"，K是"我能提供什么"，V是"我的实际内容"
  • QK点积 → softmax归一化 → 得到注意力权重 → 对V加权求和
  • √d_k作用：防止点积过大导致softmax梯度消失
  • 多头注意力：多个注意力头关注不同子空间的信息

解决的问题：RNN不能并行计算 + 长距离依赖问题

▾ Q2：大模型如何生成答案（自回归生成）

LLM采用自回归方式（Autoregressive）逐token生成：

  1. 输入经过 分词→Embedding→多层Transformer
  2. 最后一层输出一个概率分布（vocabulary size维度）
  3. 通过采样策略选出下一个token
  4. 拼到输入末尾，再次输入模型预测下一个
  5. 循环直到生成结束符（EOS）或达最大长度

    6. 核心：一个token一个token往外"蹦"的，不是一次性输出整句话

▾ Q3：Tokenization（分词过程）

  • 文本 → Tokenization（分词） → token序列
  • 主流算法：BPE（Byte Pair Encoding），将文本切为子词单元
  • 每个token映射为整数ID → Embedding层转为高维向量
  • 加上位置编码（Positional Encoding） → 输入Transformer

示例："running" → "run" + "ning"（两个子词token）

▾ Q4：Embedding模型

将文本映射为固定维度的高维向量，使语义相似的文本距离更近。

  • 常见模型：OpenAI text-embedding-ada-002、BGE、M3E、Jina
  • 核心用途：语义检索（RAG核心）、文本相似度、聚类、分类
  • 和LLM区别：Embedding只负责"理解"输出向量，不负责"生成"

▾ Q5：xxB参数含义

  • B = Billion（十亿），7B = 70亿参数
  • 参数包括所有权重矩阵中的可学习数值
  • 参数越大 → 容量越大，理论上能力上限越高
  • 但推理成本更高、延迟更大
  • PM选型视角：需要平衡性能、延迟和成本

Q13：过拟合与欠拟合

  • 过拟合：模型在训练集上表现好但测试集上差。解决：正则化、Dropout、增加数据、早停
  • 欠拟合：训练集和测试集都差。解决：增加特征、增加模型复杂度、减少正则化

Q14：模型评估指标

指标含义公式
准确率(Accuracy)预测正确的比例(TP+TN)/(TP+TN+FP+FN)
精确率(Precision)预测为正中真正为正的比例TP/(TP+FP)
召回率(Recall)真正为正中被预测为正的比例TP/(TP+FN)
F1精确率和召回率的调和平均2×P×R/(P+R)

何时看哪个：

  • 宁可漏判不可误判（如垃圾邮件）→ 重视Precision
  • 宁可误判不可漏判（如疾病筛查）→ 重视Recall

Q15：梯度下降

  • 通过计算损失函数对参数的梯度，沿梯度反方向更新参数，逐步最小化损失
  • 学习率：步长大小。太大→震荡不收敛，太小→收敛慢
  • 变体：SGD（单样本）、Mini-batch SGD（小批量）、Adam（自适应学习率，最常用）

▾ 一、RAG全流程

离线阶段（建库）

原始文档 → 清洗 → 分块(Chunking) → Embedding → 向量数据库

1. 分块（Chunking）三个核心参数：

参数含义典型值你的项目
chunk_size每块大小256-1024 tokens按标题/段落切分
overlap相邻块重叠量chunk_size的10-20%⚠️ 一面不知道，已补齐
分块策略固定长度/按分隔符/语义/递归按文档类型选按Markdown标题分隔

Overlap为什么重要：防止关键信息被切断在两个chunk的边界上。例如"内收5mm"被切在两个chunk中，单独看都不完整。

2. Embedding向量化：每个chunk通过Embedding模型转为高维向量，存入向量数据库（FAISS/Milvus/Chroma）

在线阶段（检索+生成）

用户query → [可选]Query改写 → 检索 → [可选]Rerank → Top-K → 拼Prompt → LLM生成

四种检索方式对比

向量检索关键词/全文检索混合检索Rerank
核心语义理解精确匹配两者结合二次精排
速度快快快慢（逐条）
位置召回召回召回排序（召回后）
你的项目✅ 使用✅ 使用✅ 混合检索❌ MVP未加

⚠️ 一面易错点：RAG检索的TopK（召回K个chunk）≠ LLM生成的Top-K（从K个概率最高token中采样），两个完全不同！

你的项目中的RAG配置

参数设置原因
检索模式混合检索（向量+全文）正畸术语需精确匹配+语义理解
TopK4（v1是6）v1过多召回导致LLM保守

▾ 二、RAG优化排查框架

效果不好时按此顺序排查：

第1层：输入质量 → ASR/OCR是否准确？
第2层：检索质量（最常出问题）
  ├ 召回不相关 → 调分块策略、换Embedding模型
  ├ 噪音太多 → 降TopK、提高Score阈值、加Rerank
  ├ 漏召回 → 升TopK、混合检索、Query改写
  └ 排序不合理 → 加Rerank、调权重
第3层：生成质量
  ├ 格式不对 → 强化Prompt结构约束
  ├ 幻觉 → 降Temperature、Prompt约束
  └ 模型能力不足 → 换模型
第4层：端到端体验
  ├ 延迟高 → 减检索量、用更快模型
  └ 成本高 → 降TopK、用小模型

▾ 补充：RAG回答不准怎么解决（面试高频题）

面试官问法："你的RAG系统回答不准，你怎么排查和解决？"

第一步：定位"不准"属于哪种类型

不准的类型典型表现根因方向
检索不到明明知识库有相关内容，但没被召回检索层问题
检索到但不相关召回的chunk和问题无关或关联度低分块/Embedding/检索策略问题
检索对了但回答错召回了正确内容，LLM回答仍然有误生成层问题（Prompt/模型）
回答有幻觉LLM编造了知识库中不存在的内容生成层问题 + 约束不够
回答不完整只回答了部分信息，遗漏关键内容检索量不够 / Prompt引导不足

第二步：按层逐一排查+解决

1. 数据层（知识库质量）

问题解决方案
原始文档质量差（OCR/ASR错误多）前置清洗流程，纠错后再入库
文档格式混乱（表格、图片、代码混排）按类型预处理，保留结构化信息
知识库覆盖不全定期补充新文档，建立知识缺口反馈机制
重复/矛盾内容去重 + 版本管理，保留最新版本

2. 分块层（Chunking）

问题解决方案
chunk太大 → 噪音多，关键信息被稀释减小chunk_size（如1024→512）
chunk太小 → 上下文断裂，语义不完整增大chunk_size，或增加overlap
关键信息被切断在两个chunk边界增加overlap（10-20%），或用语义分块
不同类型文档用同一分块策略按文档类型定制策略（Markdown按标题、PDF按段落、代码按函数）

你的项目实战：正畸知识库按Markdown标题分块，每个术语/场景独立成chunk，避免信息混杂。

3. 检索层（Retrieval）

问题解决方案
语义相似但词汇不同导致漏召回加入Query改写/扩展（把"牙套"扩展为"矫治器、隐适美"）
专业术语/缩写匹配不到混合检索（向量+关键词），关键词保证精确匹配
召回太多噪音降低TopK、提高Score阈值、加Rerank模型二次精排
重要chunk排序靠后加Rerank（如Cohere Rerank、bge-reranker），按语义相关度重排
用户query太模糊Query改写：用LLM先把模糊问题改写为精确检索query

检索优化组合拳（推荐顺序）：

混合检索（向量+全文） → Query改写 → 调TopK和阈值 → 加Rerank → 换更好的Embedding模型

问题解决方案

▾ 补充：多Agent系统与协作模式（面试高频题）

什么是多Agent系统

单个Agent能力有限（上下文窗口、单一角色），当任务足够复杂时，需要多个专业化Agent协作完成。类比公司组织：CEO不可能一个人完成所有工作，需要产品、研发、销售等角色分工配合。

五种核心协作模式

1. 串行（Sequential / Pipeline）

Agent A（意图识别）→ Agent B（信息检索）→ Agent C（生成回答）→ Agent D（质检审核）
  • 特点：上游输出是下游输入，按顺序执行
  • 优点：逻辑清晰、易调试、每个Agent职责单一
  • 缺点：总延迟 = 各环节之和，任一环节出错会阻塞后续
  • 适用场景：流水线式任务（文档处理、数据ETL、内容审核）
  • 你的项目类比：ASR转写 → 文本清洗 → RAG检索 → LLM生成，就是串行流水线

2. 并行（Parallel）

                ┌→ Agent A（分析维度1）→┐
用户query → 路由 ├→ Agent B（分析维度2）→├→ 结果合并
                └→ Agent C（分析维度3）→┘
  • 特点：多个Agent同时执行独立子任务，最后汇总
  • 优点：大幅降低总延迟（取决于最慢的Agent）
  • 缺点：需要合并策略，Agent之间不能有依赖
  • 适用场景：多维度分析、多源检索、批量处理
  • 举例：同时用3个Agent分别分析用户的运动数据、饮食数据、睡眠数据，最后合并成健康报告

3. 路由 / 分发（Router / Dispatcher）

                    ┌→ 售前Agent（产品咨询）
用户query → 路由Agent ├→ 售后Agent（问题处理）
                    ├→ 技术Agent（技术支持）
                    └→ 闲聊Agent（兜底）
  • 特点：一个"调度员"Agent根据意图分发给不同专业Agent
  • 优点：每个Agent可深度专业化，互不干扰
  • 缺点：路由准确性是瓶颈，误分发会导致体验差
  • 适用场景：智能客服、多业务入口、意图分类后分流
  • 关键设计点：路由Agent的Prompt要覆盖所有分支的判断规则，且需要兜底策略

4. 层级 / 主从（Hierarchical / Manager-Worker）

Manager Agent（任务拆解+调度+质检）
    ├→ Worker Agent 1（执行子任务1）
    ├→ Worker Agent 2（执行子任务2）
    └→ Worker Agent 3（执行子任务3）
Manager Agent（汇总+决策+输出）
  • 特点：有一个"经理"Agent负责拆解任务、分配给"工人"Agent、审核结果
  • 优点：处理复杂开放式任务、可动态调整策略
  • 缺点：Manager质量决定天花板，多次LLM调用成本高
  • 适用场景：复杂项目管理、多步骤研究任务、代码生成+review
  • 举例：用户说"帮我做一个竞品分析报告"，Manager拆为"收集数据→分析功能→比较定价→写报告"，分别派给不同Worker

5. 辩论 / 协商（Debate / Negotiation）

Agent A（正方观点）⇄ Agent B（反方观点）⇄ Agent C（裁判/综合）
  • 特点：多个Agent提出不同观点，互相质疑和完善，最终由裁判Agent综合
  • 优点：适合需要多角度思考的问题，输出质量高
  • 缺点：多轮交互延迟大、token消耗高
  • 适用场景：方案评审、风险评估、内容审核
  • 举例：投资决策Agent系统——乐观Agent和悲观Agent分别分析，风控Agent最终裁决

    • LangChain：通用LLM应用框架，"什么都能做"

  4. 应用层执行函数，拿到结果
  5. 结果返回LLM → 生成最终回答

▾ Q1：假设检验

流程：提出H₀/H₁ → 选统计量 → 定α(通常0.05) → 算p值 → 决策

举例：H₀:新版AI教练留存率=老版 → p=0.03<0.05 → 拒绝H₀ → 新版显著提升

▾ Q2：p值

严谨定义：在H₀为真的前提下，观察到当前结果或更极端结果的概率

常见误区：p值不是H₀为真的概率；p值小≠效果大（统计显著≠实际显著）

产品补充：大样本中微小差异也能p<0.05，所以还要看效果量和置信区间

▾ Q3：判断A/B实验策略效果（四步法）

  1. 实验设计：定核心指标+护栏指标，算最小样本量，随机分流
  2. 数据收集：跑够时间/样本量，不要中途偷看决策
  3. 统计判断：p<0.05 + 效果量有业务意义 + 护栏指标无恶化
  4. 决策：全量/调整/下线

▾ Q4：累计CTR vs 去重CTR

  • 累计CTR：总点击/总曝光，同一用户贡献多次，违反独立性假设
  • 去重CTR：点击用户数/曝光用户数，每个用户只计一次，满足i.i.d.
  • A/B测试中应用去重CTR（用户粒度）

▾ Q6：指标异动阈值

  1. 历史统计法：μ ± 2σ（95%）或 ± 3σ（99.7%）
  2. 环比/同比法：日环比下降>10%告警
  3. 业务经验法：和业务定义底线（如响应延迟>5秒告警）

    4. 结合你得物经验：通过统计方法设阈值，异动后多维下钻归因

▾ Q7：第一类/第二类错误

  • α错误（假阳性）：没效果但认为有效果
  • β错误（假阴性）：有效果但没检测到
  • Power = 1-β ≥ 0.8
  • 互联网更怕α错误（上线无用功能浪费资源）

▾ Q8：样本量确定

取决于：α(0.05) × Power(0.8) × MDE(最小有意义提升) × 基线方差

简化公式：n ≈ 16 × p(1-p) / MDE²

Keep例：留存率40%，检测2%提升 → 每组约9600人

        duration = int(duration)
        is_a = name in aerobic
        is_an = name in anaerobic
        if is_a and is_an: result["有氧+无氧"] += duration
        elif is_a: result["有氧"] += duration
        elif is_an: result["无氧"] += duration
        else: result["未知"] += duration
    return result

▾ SQL必会题型

1. 连续打卡天数（面经原题）

WITH ranked AS (
    SELECT user_id, checkin_date,
        DATE_SUB(checkin_date, INTERVAL ROW_NUMBER() OVER (
            PARTITION BY user_id ORDER BY checkin_date) DAY) AS grp
    FROM checkin
),
streaks AS (
    SELECT user_id, grp, COUNT(*) AS days
    FROM ranked GROUP BY user_id, grp HAVING COUNT(*) >= 3
)
SELECT user_id, MAX(days) AS max_days
FROM streaks GROUP BY user_id ORDER BY user_id;

原理：连续日期减行号得到相同值 → 分组 → 计数

▾ 一、Prompt Engineering系统方法论

1. Prompt的五要素结构

一个高质量的Prompt应包含：

要素说明示例
角色定义告诉模型它是谁"你是口腔正畸行业的通话记录分析专家"
核心任务明确要做什么"请根据通话转写文本生成结构化总结"
约束条件边界和限制"不得跳过任何字段，禁止'无法判定'"
输出格式期望的输出结构"必须包含四个模块：问题描述、处理描述..."
示例（Few-shot）输入→输出的参考样例提供1-2个优质分析样例

2. 核心Prompt技巧

技巧原理适用场景
Zero-shot不给示例，直接描述任务简单、通用任务
Few-shot给1-3个示例格式要求严格、领域特殊
CoT（Chain of Thought）要求模型"一步步思考"复杂推理、数学、逻辑判断
ReAct思考(Reason)+行动(Act)交替Agent任务编排
ToT（Tree of Thought）探索多个推理路径需要创造性/多方案比较
Self-Consistency多次生成取多数投票提高答案可靠性

3. Prompt调试方法论（面经原题）

"怎么确定是Prompt问题还是模型问题"：

Step1: 换模型测试（同一Prompt跑多模型）
  → 所有模型都不行 = Prompt问题
  → 只有某个模型不行 = 模型能力问题

Step2: 简化任务（复杂任务拆成小步骤测试）
  → 定位具体哪一步失败

Step3: 检查Prompt本身
  → 指令是否有歧义？是否矛盾？上下文是否过长？

Step4: 看错误模式
  → "有时对有时错" = Prompt边界不清 / Temperature太高
  → "稳定地错" = 模型能力上限 / Prompt逻辑有误

Step5: 用COT让模型输出推理过程
  → 看它在哪一步出了问题

4. 你的项目中的Prompt优化实例

v1问题归因v2优化

▾ 二、MCP协议（2026年最热概念）

什么是MCP

MCP（Model Context Protocol）= AI世界的USB-C接口

  • 由Anthropic开源，2025年底捐赠给Linux基金会
  • OpenAI、Google、Microsoft、AWS为联合创始成员
  • 统一了大模型与外部工具/数据源的连接标准
  • 2026年初月下载量突破9700万次

解决什么问题

之前：每集成一个新工具都要写自定义代码（N个模型 × M个工具 = N×M种适配）

现在：MCP统一接口，一次适配到处可用（N+M种适配）

三层架构

Host（应用层，如Cursor/Claude Desktop）
  ↓
Client（协议桥梁，管理连接）
  ↓
Server（业务逻辑封装，如数据库/API/文件系统）

三大核心原语

原语说明类比
Prompts可复用的提示模板函数模板
Resources结构化只读数据源GET接口
Tools可执行函数POST接口

面试中怎么答"你了解MCP吗？"

"MCP是Anthropic开源的标准协议，本质是统一了大模型和外部工具的连接方式，类似USB-C之于硬件。之前每个模型对接每个工具都要写定制代码，现在通过MCP标准化后，工具只需实现一次MCP Server，就能被任何支持MCP的模型调用。它的三层架构是Host→Client→Server，三大原语是Prompts、Resources和Tools。2026年它已经成为行业标准，OpenAI、Google等都加入了。对于AI PM来说，MCP的意义是大幅降低了Agent的工具集成成本，让我们可以更专注于产品逻辑而非底层适配。"

▾ 补充：Context Engineering（上下文工程）

2025年由Shopify CEO Tobi Lütke提出，迅速成为AI工程核心概念。JD中"Context Engineering"指的就是这个。

什么是Context Engineering

一句话定义：Context Engineering是为LLM构建最优上下文的系统工程——决定在每次LLM调用时，传入哪些信息、以什么结构传入、顺序如何安排。

与Prompt Engineering的关系：

Prompt EngineeringContext Engineering
范围单条Prompt的措辞和结构整个上下文窗口的信息架构
关注点"怎么问""给模型看什么"
层次微观（一个Prompt）宏观（系统级）
类比写好一个问题为学生准备考前复习资料包

Prompt Engineering是Context Engineering的子集。Context Engineering更全面——它涵盖了系统级的信息供给策略。

上下文窗口的构成

LLM的上下文窗口（Context Window）中通常包含以下信息：

┌─────────────────────────────────────────────────┐
│                Context Window                    │
│                                                  │
│  ┌────────────────────────────────────────────┐ │
│  │ 1. System Prompt（角色+规则+约束）          │ │
│  ├────────────────────────────────────────────┤ │
│  │ 2. Few-shot Examples（示例）               │ │
│  ├────────────────────────────────────────────┤ │
│  │ 3. Retrieved Context（RAG检索结果）        │ │
│  ├────────────────────────────────────────────┤ │
│  │ 4. Tool Results（工具调用返回结果）         │ │
│  ├────────────────────────────────────────────┤ │
│  │ 5. Conversation History（对话历史）        │ │
│  ├────────────────────────────────────────────┤ │
│  │ 6. User Query（当前用户输入）              │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘

Context Engineering就是优化这6个部分的内容、比例和排列顺序。

核心技术手段

手段解决什么问题具体做法
Write（写入）重要信息持久化将关键信息存入长期记忆/数据库，而非丢在对话流中
Select（选择）上下文窗口有限，不能什么都塞RAG检索、记忆过滤、对话历史压缩——只选最相关的信息
Compress（压缩）信息太多塞不下对话历史摘要化、长文档提取关键信息、去除冗余
Isolate（隔离）不同任务需要不同上下文多Agent架构中，每个Agent只看到自己需要的上下文
Order（排序）LLM对信息位置敏感最重要的信息放在开头和结尾（"Lost in the Middle"问题）
Format（格式化）结构影响理解效果Markdown/XML/JSON等结构化格式 > 纯文本

"Lost in the Middle" 问题

研究发现LLM对上下文中开头和结尾的信息关注度最高，中间部分容易被忽略。

注意力分布：
高 ████████░░░░░░░░░░░░░░░░░░░░░░████████  高
   ↑开头                    中间↑         ↑结尾

应对策略：

  • 最关键的规则放在System Prompt开头
  • RAG检索结果按相关度排序，最相关的放最前
  • 用户query放在最后（靠近生成位置）
  • 长上下文中插入"提醒"标记，防止遗忘

技术手段你的项目实践

面试回答模板

"Context Engineering是比Prompt Engineering更高一层的概念。Prompt Engineering关注的是'怎么问'，Context Engineering关注的是'给模型看什么'——在有限的上下文窗口中，如何选择、组织、排列信息，让LLM获得最优的理解和生成效果。

核心手段包括：Write（持久化关键信息）、Select（RAG检索最相关内容）、Compress（压缩冗余信息）、Order（利用Lost in the Middle效应合理排序）、Format（结构化格式提升理解）、Isolate（多Agent场景下隔离上下文）。

在我的项目中，我通过RAG的TopK和Score阈值控制Select，通过噪音过滤规则实现Compress，通过Markdown模板实现Format，通过知识库实现Write。这些本质上都是Context Engineering的实践。"

OpenAPI（Swagger）

是什么

OpenAPI是一种RESTful API的标准描述格式（YAML/JSON），告诉调用者：这个API有哪些接口、每个接口需要什么参数、返回什么数据。

类比：OpenAPI是API的"说明书"，就像餐厅菜单——列出所有菜品、价格、配料，顾客看菜单点菜。

核心结构

openapi: 3.0.0
info:
  title: 正畸病例管理API
  version: 1.0.0
paths:
  /patients/{id}:
    get:
      summary: 获取患者信息
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  name: { type: string }
                  diagnosis: { type: string }
                  stage: { type: string }

在Agent系统中的作用

  1. 工具注册：Agent通过读取OpenAPI文档，自动了解可用工具的功能和参数
  2. Function Call的基础：LLM的Function Calling本质上就是根据类似OpenAPI的Schema来生成调用参数
  3. 自动化集成：有了OpenAPI文档，可以自动生成SDK、测试用例、文档
  4. MCP Server的底层：很多MCP Server的Tools实际上是对OpenAPI接口的封装

    概念说明举例

    RPC / gRPC

    是什么

    RPC（Remote Procedure Call）= 远程过程调用，让调用远程服务像调用本地函数一样简单。gRPC是Google开源的高性能RPC框架。

    类比：OpenAPI像发邮件（写清楚格式、等回复），RPC像打电话（直接说，实时响应）。

    gRPC的核心特点

    特点说明为什么重要
    Protocol Buffers二进制序列化格式（比JSON小3-10倍）高性能，低带宽
    HTTP/2多路复用、服务端推送支持双向流式通信
    强类型用.proto文件定义接口和数据类型编译时校验，减少运行时错误
    多语言自动生成10+种语言的客户端代码异构系统集成
    流式通信支持单向流和双向流适合实时数据传输

    Proto文件示例

    service PatientService {
  rpc GetPatient (PatientRequest) returns (PatientResponse);
  rpc StreamUpdates (PatientRequest) returns (stream UpdateEvent);
}

message PatientRequest {
  string patient_id = 1;
}

message PatientResponse {
  string name = 1;
  string diagnosis = 2;
  string stage = 3;
}

    gRPC的四种通信模式

    模式说明适用场景
    Unary一请求一响应查询患者信息
    Server Streaming一请求多响应（服务端持续推送）LLM流式输出
    Client Streaming多请求一响应（客户端持续发送）批量上传数据
    Bidirectional Streaming双向持续通信实时对话、语音交互

    在Agent系统中的作用

    场景为什么用gRPC
    模型推理服务大量并发调用，需要高性能（Triton等推理服务都用gRPC）
    Agent间通信多Agent系统中Agent之间需要低延迟通信
    流式输出LLM逐token生成时，用Server Streaming实时推送
    微服务架构后端各服务间高性能调用

    OpenAPI vs RPC vs MCP：选型指南

    场景推荐协议原因
    对外提供API给第三方OpenAPI标准化、易理解、生态好
    内部微服务通信gRPC高性能、强类型
    AI模型调用外部工具MCPAI原生、自动发现、标准化
    LLM流式输出gRPC (Server Streaming) 或 SSE支持逐token推送
    Agent注册新工具MCP + OpenAPIMCP注册，底层可能调用OpenAPI接口

    三者的协作关系

    用户 → Agent（MCP Client）→ MCP Server → OpenAPI接口 / gRPC服务 → 后端系统
                                ↑
                          MCP统一了"注册和发现"
                     OpenAPI/gRPC负责"实际调用"

    实际工程中三者经常搭配使用：

    • MCP 负责工具的注册、发现和权限管理
    • OpenAPI 描述HTTP接口的参数和返回格式
    • gRPC 负责高性能场景下的实际数据传输

    面试回答模板

    "这三种协议在Agent系统中各有分工：OpenAPI是API的标准描述格式，定义了接口有哪些、参数是什么、返回什么，它让Agent能'读懂'一个工具的说明书；gRPC是高性能的远程调用协议，用二进制+HTTP/2实现低延迟通信，适合模型推理和Agent间通信；MCP是AI原生的连接标准，统一了模型与工具的注册和发现。

    三者的协作关系是：MCP负责工具的注册和发现，OpenAPI描述接口格式，gRPC负责高性能调用。在我的项目中，Dify的知识检索节点本质上就是通过类似OpenAPI的方式调用向量检索服务，LLM的Function Call也是基于类似OpenAPI Schema来生成调用参数的。"

graph = StateGraph(AgentState)

graph.add_edge("classifier", "retriever")

▾ 四、Prompt Engineering面试高频题补充

Q1：CoT和ReAct的区别？

- CoT：让模型在回答前先"思考"，输出推理链。适合数学、逻辑推理

- ReAct：思考和行动交替进行——思考(需要查天气) → 行动(调用天气API) → 观察(结果是晴天) → 思考(可以推荐户外跑步) → 行动(生成方案)。适合需要调用工具的Agent场景

Q2：如何设计一个好的System Prompt？

1. 明确角色和专业领域

2. 定义任务边界（做什么、不做什么）

3. 输出格式强约束（用模板而非自然语言描述）

4. 加入关键的负面指令（"不要..."、"禁止..."）

5. 提供1-2个优质示例（Few-shot）

6. 设定兜底策略（不确定时的默认行为）

Q3：Prompt版本管理怎么做？

1. 每个版本的Prompt存档（你的项目有v1/v2）

2. 记录每次改动的原因和效果

3. 用同一批测试数据做AB对比

4. 建立Prompt变更日志：改了什么 → 解决了什么问题 → 指标变化

▾ 五、你的Prompt迭代实战案例（企微聊天版）

面试时用自己的真实案例讲Prompt迭代，比背方法论有说服力100倍

背景

将AI摘要系统从通话场景扩展到企微群聊场景，用真实聊天记录测试后发现多个问题。

v1→v2迭代过程

问题1：多病例遗漏

现象：一段群聊讨论了3个患者（易伟杨、袁茹、辛晓煌），v1 Prompt跑3次，有1次只输出了2个患者的分析

归因：LLM倾向于关注信息密度高的病例，"易伟杨"只有一问一答2条消息，容易被跳过

优化：在Prompt中新增强制的"预扫描"步骤：

## 第一步：扫描并列出所有患者
在开始分析之前，你必须先完整扫描全部聊天记录，找出所有被提及的患者姓名。
扫描规则：
- 即使某个患者只被提到一次、只有一问一答，也必须列出

效果：一致性从66%（2/3）→ 100%（3/3）

Prompt设计原理：利用LLM的"先规划后执行"特性，强制做全局扫描再逐个分析，避免遗漏

问题2：转述场景丢失

现象：客服转述医生意见（"临床说：这个患者..."），v1没有提取到医生的判断

优化：新增角色识别规则：

- **客服/中间人**：名称中含"客服"、转述他人意见（如"临床说..."、"麻烦告知X医生..."）
特殊规则：如果医生意见是通过他人转述的，仍然基于转述内容分析医生的性格特征，但标注"（通过XX转述）"

问题3：噪音干扰

现象：群聊中大量的系统消息、表情、寒暄影响分析质量

优化：明确过滤规则（白名单思维 > 黑名单思维）：

忽略以下内容：
- 群创建信息、系统消息
- 客服的流程性消息（如"收到"、"已转达"）
- 质量管理、产品管理等非病例讨论内容
- 纯表情、纯寒暄

问题4：术语纠错扩展

现象：真实聊天中出现通话版未覆盖的错别字

优化：基于真实数据补充纠错规则：

- "桥治" → "矫治"
- "丢抗" → "丢失支抗/失抗"

一致性测试方法论

同一输入跑3次，对比输出的：

  1. 信息覆盖一致性：是否每次都识别到相同的患者/问题
  2. 判断一致性：性格判定、风格评级是否基本一致
  3. 结构一致性：输出格式是否每次都符合模板

    4. 关键认知

    1. Prompt迭代必须由数据驱动：不是"我觉得该改什么"，而是"真实数据告诉我哪里有问题"
    2. 白名单优于黑名单：与其列出所有不要的内容，不如明确只提取什么
    3. 强制步骤>建议步骤：LLM会偷懒，"必须先..."比"请尝试..."有效
    4. 一致性测试是AI产品的基本功：单次结果好不代表稳定好

    Q4：你怎么理解ICL（In-Context Learning）？在项目中怎么用的？

    答：

    定义：ICL（In-Context Learning）是指不需要微调模型参数，而是通过在Prompt中提供示例（Few-shot），让模型在推理时"现学现用"的技术。本质上是利用大模型的上下文理解能力，通过样例来引导输出。

    和相关概念的关系：

    • Zero-shot：不给示例，直接让模型完成任务（效果最差但最灵活）
    • Few-shot / ICL：给2-5个示例，让模型理解期望的输入-输出模式（效果和成本的平衡点）
    • 微调（Fine-tuning）：用大量数据训练模型参数（效果最好但成本最高、不灵活）

    我在项目中的具体应用：

    1. 正向样例引导：在Prompt模板中嵌入2-3个"优质分析样例"——包含一段通话文本和对应的高分摘要输出，让模型理解"好的摘要长什么样"
    2. 反面样例约束：加入bad case样例并标注"这种输出是错误的，因为遗漏了第二个患者的信息"，告诉模型什么不能做
    3. 动态更新：迭代过程中不断用最新的高分输出替换旧样例，保持样例的代表性
    4. 场景化ICL：不同通话场景（初诊/复诊/投诉）使用不同的示例集，提高针对性

      5. ICL在内容安全的应用：可以在Prompt中给出"违规/合规"的成对样例，让模型学习识别隐晦违规表达（如谐音梗、隐喻、暗语）——给几个已标注样例，模型理解模式后就能识别新变体。

      第二类：内容安全业务理解（概率高，占20%）

      Q5：RAG/ICL在内容安全场景怎么应用？

      答：

      RAG的应用：

      • 当新内容需要审核时，用RAG检索历史违规案例库和最新政策文件
      • 将检索到的相似违规案例和对应政策条款作为上下文注入Prompt
      • 让大模型基于"历史案例 + 政策依据"来判断当前内容是否违规，并给出违规原因和政策依据
      • 核心优势：政策更新时只需更新知识库，无需重新训练模型——内容安全政策经常变化，RAG比微调更适合

      ICL的应用：

      • 在Prompt中给出违规/合规的成对样例（Few-shot）
      • 特别适合识别隐晦违规——比如谐音梗"s1"代替敏感词、用隐喻暗示违规内容
      • 给几个已标注样例，模型学习"这种表达模式=违规"后就能识别新变体

      RAG + ICL结合：

      新内容输入 → RAG检索相似历史案例 → 将案例作为Few-shot样例注入Prompt → 模型基于样例+政策判断是否违规 → 输出判断结果+置信度+违规依据

      和我项目的类比：

      • 我的项目：检索通话场景知识 → 注入Prompt → 模型生成通话摘要
      • 内容安全：检索违规案例+政策 → 注入Prompt → 模型判断是否违规
      • 方法论完全一致，只是应用场景不同

      Q7：你怎么理解"准确率"和"召回率"在内容安全场景的权衡？

      答：

      定义：

      • 准确率（Precision）：模型判定为违规的内容中，真正违规的比例 → 高准确率 = 少误杀
      • 召回率（Recall）：所有真正违规的内容中，被模型成功识别的比例 → 高召回率 = 少漏杀

      内容安全的特殊权衡：

      • 内容安全通常召回率优先——漏掉一条违规内容可能引发重大舆情事故，但误杀一条合规内容的代价相对较小（用户可申诉恢复）
      • 但也不能无限追求召回率——误杀率太高会严重损害创作者体验和平台生态
      • 实际策略：分级处理——高风险类型（涉政/涉恐）召回率优先，低风险类型（擦边/软广）准确率和召回率平衡

      和我项目的关联：我在AI摘要项目中也面临类似权衡——"信息完整性"（类似召回率，不漏关键信息）和"准确性"（类似准确率，不编造信息）之间的平衡。我的解决方案是在评估体系中给"完整性"更高权重（40分 vs 30分），同时用负向约束控制幻觉。

      Q8：简述Transformer架构和Self-Attention机制

      答：

      Transformer架构：

      Transformer由Encoder和Decoder两部分组成：

      • Encoder：将输入序列编码为上下文相关的特征表示。每层包含Self-Attention + Feed-Forward Network
      • Decoder：基于Encoder输出和已生成的token，自回归地生成下一个token。包含Masked Self-Attention + Cross-Attention + FFN

      当前主流LLM（GPT系列）只用Decoder，原因：

      • 因果注意力（Causal Attention）天然适合自回归生成任务
      • 训练时每个位置都可以作为预测目标，训练效率更高
      • 结构更简单，易于扩展到超大规模

      Self-Attention机制：

      输入 → 通过三个权重矩阵生成Q（查询）、K（键）、V（值）
→ 计算 Attention(Q,K,V) = softmax(QK^T / √d_k) × V
      • Q（Query）：当前token想要查询的信息
      • K（Key）：每个token提供的"索引标签"
      • V（Value）：每个token携带的实际信息
      • QK^T：计算当前token和所有token的相关性分数
      • 除以√d_k：防止点积值过大导致softmax梯度消失（d_k是Key的维度）
      • softmax：将分数归一化为概率分布
      • ×V：用注意力权重加权求和，得到融合了上下文信息的表示

      Multi-Head Attention：将Q/K/V分成多个头，每个头关注不同的语义维度（如语法关系、语义关系、位置关系），最后拼接起来。

      Q9：大模型幻觉的原因和解决方案

      答：

      产生原因：

      1. 训练数据噪声：预训练语料中包含错误或过时信息
      2. 概率采样的随机性：Temperature > 0时，模型按概率分布采样，可能选到低概率但不正确的token
      3. 模型过度泛化：模型学到了"看起来合理"的模式，但缺乏事实性判断能力
      4. 知识截止：模型训练数据有时间截止，无法获取最新信息

        5. 解决方案：

        1. RAG（检索增强生成）：用检索到的事实文档来约束生成，让模型"有据可查"
        2. ICL（Few-shot样例）：通过正确样例引导输出方向，减少偏离
        3. 降低Temperature：减少采样随机性，使输出更确定（但牺牲多样性）
        4. 负向约束Prompt：明确告诉模型"不要编造未提及的信息"
        5. 后处理校验：对模型输出进行规则校验或二次模型审核
        6. 人工复审兜底：低置信度结果进入人工审核

          7. 我的实战经验：在AI摘要项目中，幻觉的主要表现是"编造治疗方案"和"给未提及的患者加信息"。我的解决方法是：RAG引入正确的场景知识 + ICL给出正确样例 + 负向约束"只提取明确提及的信息" + 评估体系中"编造"定义为致命错误。

          Q10：RAG和微调（Fine-tuning）怎么选？

          答：

          对比维度RAG微调（Fine-tuning）
          适用场景知识频繁更新、需要引用来源风格/格式固化、特定领域深度适配
          更新成本低——只需更新知识库高——需要重新训练
          时效性强——知识库可实时更新弱——训练数据有截止日期
          可解释性强——可追溯到检索的原始文档弱——模型内部是黑箱
          幻觉控制较好——有事实文档约束一般——仍可能产生幻觉
          计算成本推理时增加检索开销训练时成本高，推理时无额外开销

          内容安全治理更适合RAG，原因：

          1. 违规标准和政策经常变化（新规定、新热点、新违规手段），RAG只需更新知识库
          2. 需要引用政策依据——审核结果要说清楚"依据哪条政策判定违规"，RAG天然支持溯源
          3. 新型违规不断涌现，RAG可以快速补充案例，微调需要重新收集数据训练

            4. 也可以RAG + 微调结合：用微调让模型更好地理解内容安全领域的语义和术语，用RAG提供最新的政策和案例。

            Q11：LoRA是什么？为什么不用全量微调？

            答：

            LoRA（Low-Rank Adaptation）：一种参数高效微调方法。核心思想是：不修改原始模型的全部参数，而是在每层Transformer中插入两个小矩阵（低秩矩阵A和B），只训练这两个小矩阵。

            原始权重W（冻结） + ΔW = W + A×B（只训练A和B）
其中A的维度是 d×r，B的维度是 r×d，r远小于d（比如r=8）

            为什么不用全量微调：

            1. 参数量：7B模型全量微调需要训练70亿参数，LoRA只需训练几百万参数（不到原来的1%）
            2. 显存：全量微调7B模型需要约56GB显存，LoRA只需约14GB
            3. 速度：训练时间大幅缩短
            4. 灾难性遗忘：全量微调容易让模型"忘记"预训练学到的通用能力，LoRA影响更小
            5. 多任务：可以为不同任务训练不同的LoRA适配器，切换时只需加载不同的A/B矩阵

              6. 第四类：行为面/价值观（概率高，占20%）

▾ 三、高频追问及回答要点

Q1："准确率90%是怎么算的？"

通话版：从100条分析结果中随机抽取50条，人工对照原始录音/聊天记录进行逐项核验——

  • 核心信息覆盖率（是否遗漏关键问题和方案）
  • 术语准确性（专业术语是否正确识别和使用）
  • 角色识别准确率
  • 结论判断准确性

加权计算综合准确率。每周抽检一批，持续监控，发现bad case立即归因并迭代Prompt。

Q2："为什么不用Fine-tune，而是用RAG+Prompt？"

四个原因：

  1. 数据量不足：正畸领域标注数据有限，Fine-tune需要数千条高质量样本
  2. 迭代速度：Prompt可以当天改当天验，Fine-tune需要重新训练
  3. 成本：Fine-tune需要GPU资源和标注人力，MVP阶段投入产出比不合理
  4. 知识更新：术语表、场景规则需要频繁更新，RAG只需更新知识库文档

    5. 如果未来数据量达到5000+条、且输出质量仍有瓶颈，会考虑LoRA微调。

    Q3："你怎么设计的评估体系？"

    我设计了三层评估机制：

    • 第一层：100分制评分表——每个样本从5个维度评分（问题描述、处理描述、性格判定、风格判定、通用质量），确保评估全面且可量化
    • 第二层：MVP达标线——A级(≥85)可上线、B级(70-84)微调后上线、C级(55-69)大改、D级(<55)重做
    • 第三层：Bad Case归因模板——每个bad case归因到四类之一（Prompt缺陷 / 知识库缺失 / 模型能力 / 输入质量），指导精准优化

    设计逻辑：评分告诉我"好不好"，达标线告诉我"能不能上线"，归因告诉我"怎么改"。

    Q4："从通话版到企微版，最大的技术挑战是什么？"

    多病例拆分。一段群聊讨论了3个患者，v1 Prompt有33%概率漏掉其中一个。分析原因：LLM倾向于关注信息密度高的病例，跳过只有1-2条对话的简短病例。

    解决方案：在Prompt中加入强制的"预扫描"步骤——先列出所有患者名单，再逐个分析。这利用了LLM的"先规划后执行"特性，效果显著（一致性从66%→100%）。

    Q5："Dify vs Python脚本，你为什么选Dify做企微版？"

    维度Python脚本Dify Workflow
    开发速度需要写代码拖拽搭建20分钟
    维护成本需要开发者业务人员可自行调参
    知识库管理自建向量库内置知识库（支持混合检索）
    可复用性需要改代码改Prompt即可适配新场景

    通话版用Python是因为需要集成ASR API（Dify不直接支持阿里云ASR）；企微版输入就是文本，Dify是更优选择。
